# BadPunPC Auto-Solver — 系统设计

> 状态：初稿（探索阶段写完后会回填具体坐标 / 交互细节）

## 1. 游戏机制（已观察）

通过 No.81 这道样题观察到：

- 题型：上下两幅对比图 + 一句完整caption + 一句带空格的caption
- 上图配文如 "这是坐在一起"（描述完整场景）
- 下图配文如 "这是____"（去掉一个元素后的场景）
- 屏幕右侧给出 4 个汉字字卡（带拼音），按正确顺序填入空格
- 类型标签：成语 / 俗语 / etc.（左下角粉色徽章）
- 通关后出现 "恭喜通关" 绿色按钮

**核心解题逻辑（人类视角）**：

```
上图描述 = 字面意思（例：坐在一起）
下图 = 上图 减去 某物体（例：减去鳄鱼）
谜底 ≈ "无 + [物体谐音字] + 不 + [上图描述谐音字]"
       例：无鳄不坐 → 谐音 → 无恶不作
```

不是所有题都遵循这个模式，但这是一个常见模板。

## 2. 系统架构

```
┌──────────────┐
│ 1. screenshot │ ← computer-use 截屏
└──────┬────────┘
       ▼
┌──────────────────────┐
│ 2. 视觉理解 + 推理     │ ← Claude 多模态原生看图
│   - 识别题号 No.X     │
│   - 提取上图/下图内容  │
│   - 提取字卡 4 个字   │
│   - 推谐音梗谜底      │
└──────┬───────────────┘
       ▼
┌──────────────────────┐
│ 3. 查 answers.jsonl   │ ← 题号已解过？直接用缓存答案
│   缓存命中 → 直接输入  │
└──────┬───────────────┘
       ▼
┌──────────────────────┐
│ 4. 查 wrong_attempts  │ ← 排除上次错的候选
└──────┬───────────────┘
       ▼
┌──────────────────────┐
│ 5. 输入答案           │
│   方式 A：键盘 type   │ ← 用户首选
│   方式 B：点击字卡    │ ← fallback
└──────┬───────────────┘
       ▼
┌──────────────────────┐
│ 6. 截屏验证反馈       │
│   - "恭喜通关" → 成功  │
│   - 其它 → 错误        │
└──────┬───────────────┘
       │成功                    错误
       ▼                          ▼
   写 answers.jsonl       写 wrong_attempts.jsonl
   找 "下一题" 按钮          回到 step 2 重推
```

## 3. 反馈机制（用户已确认 — Wordle 式）

每次提交答案后，游戏对每个位置的字（拼音 + 汉字）独立着色：

| 颜色 | 含义 |
|---|---|
| 🟢 绿色 | 拼音 / 汉字都正确，位置也对 |
| 🟠 橙色 | 拼音存在于答案中但位置错（汉字同理独立判定） |
| ⚪ 灰/默认 | 不在答案里 |

**意味着**：
- 第一枪即使全错，也能通过颜色信息推断正确位置 / 出现的字
- 拼音和汉字是**两个独立通道**，各自上色 → 可以用拼音相同但字不同的同音字试探（典型谐音梗题，正确答案的拼音 = 字面意思的拼音）
- 给定字数（一般 4 字），最多需要 2~3 枪即可定位

**新策略**（替换原来的"记录错误候选"）：

```
第 1 枪：根据视觉推理出"字面句子"对应的成语 → 试这个
  例：看到 "无鳄不坐" → 第 1 枪输入 "无恶不作"（直接谐音猜）
  若全绿 → 通关
若有橙/灰 → 解析颜色信息：
  - 锁定绿色位置不动
  - 橙色位置：拼音存在但需换位 → 在剩余位置中重排
  - 灰色位置：拼音不在 → 换字
  - 用 pinyin 库枚举同音字候选 → 选最合理成语第 2 枪
```

## 3a. 探索结果（已实测）

### 启动前置
- 必须给游戏加 Steam 启动选项 `--disable-gpu`，否则截屏全黑（GPU 直渲染绕过截屏 API）
- 必须 request_access 同时授权：`BadPunPC.exe` + `msedgewebview2.exe`（实际渲染进程）

### 输入与提交
- **输入方式**：键盘直接 type 中文字符（computer-use 走 clipboard fast path），不需要中文 IME 切换
- **输入框位置**：右下区域，占位符「输入X字词语」（X 随题目变化）
- **答题前需 left_click 输入框**让其获得焦点
- **预览效果**：打字时上方空格框实时显示候选汉字 + 自动标拼音
- **提交方式**：按 `Return` (Enter)；或 left_click 「确定」按钮

### 反馈机制（Wordle 式）— 已确认绿色
- **绿色**：拼音/汉字 + 位置全对 → 同时出现「恭喜通关」+「点赞/点踩」按钮
- 橙色 / 灰色：**还没观察到，下次故意答错时验证**

### 题目格式
- 类型 badge 在左下：`成语 / 电影术语 / 俗语 / ...` 类型决定答案字数
- No.82 类型 = 电影术语，2 字
- No.81 类型 = 成语，4 字
- → **答案字数从输入框空格数读取，不是固定的 4**

### 关卡切换
- 通关后看到「恭喜通关」绿按钮，**点它推测进下一题**（待实测）

### 提示功能
- 右下角灯泡图标 + 「TAB」标签 → 按 TAB 键给提示（先不用，避免污染数据）

### 题号唯一性
- 题号 No.X 可作主键，配合左侧 caption 哈希做二次校验防止题库改动

## 3b. 第二阶段实测补充（No.83 油麦菜）

### 错误反馈（已确认 灰色）
- 提交后**整张拼音+汉字小卡片**上移成"历史尝试"行
- 整卡 = 灰色背景 + 黑字 + 黑拼音 → 该字 / 该拼音都不在答案
- 进度条左侧黄色小圆 += 1（计数已用错误次数）
- 输入框自动清空，光标保留焦点 → 可直接打下一次答案

### 橙色（仍未实测）
- 推断：单字格 = 橙色背景 / 黄色背景 → 拼音或字命中但位置错
- 拼音和汉字独立判定（同一格可能"拼音绿+字橙"或"拼音灰+字橙"等组合）
- **下次实战自然遇到时立即截屏归档**

### 提示功能（TAB）
- TAB 切换到提示模式：底部按钮变黄色「提示↵」
- 按 Enter / 点提示按钮 → 揭示下一个未填位置的**先拼音再字符**
  - 第一次按：揭示第 1 位的拼音
  - 第二次按：揭示第 1 位的字
  - 第三次按：揭示第 2 位的拼音
  - ……
- 揭示出的字 = 绿底（视觉上等同正确）
- **再按 TAB 退出提示模式 → 揭示的字消失**（不会持久填进答案，必须自己完整输入提交）
- 进度点不增加 → **提示不算 wrong attempt**
- 但有可能有"提示次数上限"或"通关成绩降级"——本次没观察到

### 通关切换
- 全绿后「恭喜通关」按钮 + 点赞/点踩 + 右下角成就弹窗
- **大约 5–10 秒后自动跳下一题**（不需要点击）
- 若在等待期间触发了别的事件（如 alt-tab、点击窗口外等），可能游戏会回到主菜单

### 主菜单恢复路径
- 主菜单：单人游戏 / 多人游戏 / 切换玩家 / 退出
- 单人游戏 → 经典题库（带"继续"黄徽章可续上次）/ 分类专题 / 创意工坊
- → 必须经过 2 层菜单才能回到当前题
- **建议**：自动模式开始前确认窗口在题目页;若发现回到主菜单,自动 click 单人 → 经典题库继续

### 窗口尺寸 / 渲染怪象
- `--disable-gpu` 后游戏窗口出现**黑色 letterbox**（上下黑带，画布只占中间）
- 鼠标坐标用相对画布位置不可靠 —— **每次操作前先截屏定位**
- 或者：用 PIL/OpenCV 找题目卡片的白色矩形锚点,再算输入框相对偏移

### Achievement 通知干扰
- 通关后右下角弹"成就解锁"小卡片(如"国"、"地" 等单字)
- 这个弹窗**不在游戏窗口内**,可能盖住其他 UI
- 等下一题加载后会消失,不需要主动关闭

## 4. 系统架构 v2(更新版)

```
init:
  确认游戏在题目页(否则导航回去)
  load answers.jsonl 到内存

solve_loop(当前题):
  screenshot
  → vision_extract:  no, top_caption, bottom_caption_diff, category, answer_length
  → if no in answers.jsonl: 用缓存答案直接 type
     elif: candidates = generate_candidates(scene, category, length)
           guess = candidates[0]
  → click input field, type guess, press Enter
  → wait 0.8s, screenshot
  → parse_feedback:
       all_green → success, append answers.jsonl, wait 8s for auto-advance, loop
       has_orange_or_gray → 用颜色信息缩小候选集,挑下一枪
  → 如果 N=5 次仍未对: TAB 一次拿提示,再做 1 次,还不对就停下报错
```

## 5. 反馈解析 — 颜色 → 约束

提交后,每个 slot 的**汉字背景 + 拼音背景**独立着色。归类成约束放进逻辑求解器:

| 字背景 | 拼音背景 | 含义 | 约束 |
|---|---|---|---|
| 绿 | 绿 | 字 + 拼音 + 位置全对 | 该位置锁死,不再变 |
| 灰 | 绿 | 拼音对位置对,字错(罕见,需同音字) | 此位换同音异字 |
| 灰 | 橙 | 拼音存在但位置不对,字错 | 此拼音放别的位置 + 换字 |
| 灰 | 灰 | 都不在答案 | 排除该字 + 该拼音整体 |

(更多组合实战遇到再补)

## 6. 候选生成 prompt(给 Claude 视觉推理用)

```
你是谐音梗解题助手。下面是 BadPunPC 当前题目截图。

观察并输出 JSON:
{
  "no": <题号>,
  "category": <左下徽章文字>,
  "answer_length": <输入框空格数>,
  "top_caption": "<上图配文(完整版)>",
  "bottom_diff": "<下图相比上图新增/修改了什么(物体/动作/对话)>",
  "scene_pun_analysis": "<推理:这个新元素的发音 / 字面意义 / 视觉双关>",
  "literal_phrase": "<把下图描述成完整 N 字短语,不一定是成语>",
  "candidates": [
    {"answer": "X字", "reason": "<为什么这是答案>", "confidence": 0-1},
    ...至少 3 个
  ]
}

约束:候选必须满足 (a) 字数 = answer_length (b) 是 category 类型 (c) 与 top_caption 形成谐音关系
```

## 7. 失败模式 / 边界保护

- 题号读不出 → 拍照存到 D:\AI做谐音梗题\debug\<ts>.png 后停下问用户
- 颜色解析二义性(光线/缩放) → 用 RGB 阈值 + 先验色卡(实测样本)
- 网络/游戏卡死 → 30s 没变化就 alt-tab 后再回来
- 字数与候选不匹配 → 不提交,记 log 让用户介入
- 同一题反复挂在橙色组合 → N=8 次后 TAB 提示

## 4. 数据格式

`answers.jsonl`：每行一题
```json
{"no": 81, "answer": "无恶不作", "category": "成语", "first_try": false, "tries": 2, "ts": "2026-05-03T02:18:00"}
```

`wrong_attempts.jsonl`：每个错误尝试一行
```json
{"no": 81, "wrong": "无鳄不坐", "ts": "..."}
```

## 5. Prompt 设计（解题阶段）

给 Claude 的指令（每次截屏后用）：

> 这是 BadPunPC 游戏当前题目截图。
> 任务：
> (a) 读出题号
> (b) 描述上图、下图差异（哪个角色/物体消失了）
> (c) 读出右侧 4 个候选字
> (d) 推断字面句子 + 谐音目标成语/俗语
> (e) 给出 4 字答案及推理过程
> 输出 JSON：{no, top_caption, bottom_diff, chips, literal, answer, reasoning}

## 6. 风险 / 边界

- 视觉识别错（图里有冷门动物 / 物品）→ 列 top-3 候选轮试
- 同名题号但实际题不同 → 缓存命中后仍校验上图 caption 是否一致
- 游戏窗口被遮挡 / 不在前台 → 每次操作前 screenshot 验证 + open_application
- 字卡的字数和成语字数对不上 → 题目可能不是 4 字成语，需要 fallback 到 N 字模板
